# 회의자료 — 현대 MoE 방법론에서 연구 문제 정의하기

> **안건**: 논문을 쓰기 위한 "현대 Mixture-of-Experts(MoE) 모델의 연구 문제" 정의
> **접근**: *measure-first* — 추측이 아니라, 실제 모델의 라우팅을 측정해서 문제를 찾는다
> **대상 모델**: Qwen3-30B-A3B (완료) + Mixtral-8x7B (교차검증 진행 중)
> **작성일**: 2026-06-15

---

## 0. 30초 요약 (바쁜 분용)

- 우리는 거대 MoE 모델이 토큰을 어느 "전문가(expert)"로 보내는지 **48개 층 전부를
  직접 측정**했다.
- 처음에 유력해 보이던 압축 아이디어 3개(중복 전문가 병합 / 활성 전문가 수 줄이기 /
  라우터 다듬기)는 **데이터로 전부 기각**됐다.
- 살아남아 **실제로 검증된 문제**: *"배포할 도메인이 정해지면, 그 도메인이 안 쓰는
  전문가의 16~22%를 정확도 손실 거의 없이 끌 수 있다. 단 어느 전문가를 끄느냐는
  도메인마다 완전히 달라서, 도메인을 잘못 잡으면 성능이 무너진다(PPL +14)."*
- → **논문 문제 정의: "도메인-조건부 전문가 가지치기(domain-conditional expert
  pruning)"**. 라우팅 측정으로 근거가 확보되고, 마스킹 실험으로 검증까지 끝남.

## 1. 배경 지식 — MoE를 모르는 분을 위한 5분 설명

### 1.1 일반 신경망 vs MoE
보통의 Transformer는 모든 입력 토큰이 **똑같은 거대한 연산 블록(FFN)** 을 통과한다.
모델을 키우려면 이 블록을 키워야 하고, 그러면 **모든 토큰이** 더 많은 계산을 한다.

**MoE(Mixture-of-Experts)** 는 이 블록을 여러 개의 작은 "**전문가(expert)**"로 쪼갠
뒤, 토큰마다 **그중 일부만** 골라서 통과시킨다.
- 예: 전문가가 128명 있는데, 토큰 하나는 그중 **8명에게만** 보낸다.
- 누구에게 보낼지 정하는 작은 부품을 "**라우터(router / gate)**" 라고 한다.
- 결과: 모델의 **총 파라미터(지식)는 크지만, 토큰 1개가 실제 쓰는 계산은 작다.**
  (Qwen3-30B-A3B = 총 30B 파라미터지만 토큰당 약 3B만 활성 = "A3B")

### 1.2 핵심 용어 (이 문서 전체에서 사용)
| 용어 | 뜻 |
|------|----|
| **expert(전문가)** | 토큰을 처리하는 작은 FFN 하나. Qwen3는 층마다 128개. |
| **router/gate(라우터)** | 토큰을 어느 전문가에게 보낼지 점수를 매기는 부품. |
| **top-k** | 라우터 점수 상위 k명에게만 토큰을 보냄. Qwen3는 top-8. |
| **routing(라우팅)** | "토큰 → 전문가" 배정. 이 연구의 측정 대상. |
| **dead expert(죽은 전문가)** | 거의 아무 토큰도 안 받는 전문가. 메모리만 차지. |
| **fingerprint(지문)** | 한 입력이 전체 층에서 어떤 전문가들을 썼는지의 패턴. |

### 1.3 왜 이게 연구 문제인가?
거대 MoE는 강력하지만 **전문가를 전부 메모리에 올려둬야** 한다(쓰든 안 쓰든).
"안 쓰는 전문가를 안전하게 버릴 수 있나?"가 곧 **모델 압축/배포 비용** 문제다.
그런데 *어떤 전문가가 정말 안 쓰이는지*는 추측이 아니라 **측정**해야 안다.
→ 그래서 우리는 측정부터 했다.

## 2. 우리가 한 일 — 실험 설정

- **모델**: Qwen3-30B-A3B = **48개 층 전부가 MoE**, 층마다 **128 전문가, top-8**.
- **데이터(3개 도메인 × 2개 데이터셋)**:
  - 수학(math): GSM8k, SVAMP
  - 코드(code): HumanEval+, MBPP+
  - 자연어추론(nli): MNLI, SNLI
  - 각 200개 예시.
- **방법**: 텍스트를 모델에 **한 번 통과(forward)** 시키면서, 48개 층의 라우터가 매
  토큰을 어느 전문가로 보내는지 **전부 기록**했다. (답을 생성하진 않음 — 라우팅만 관찰)
- 측정 가설은 두 라운드로 나눴다: **H1~H6**(라우팅의 구조), **P1~P6**(약점/기회),
  그리고 후속 정밀실험 **E1~E3**.

> ※ 가설 이름의 H/P/E는 "Hypothesis(가설) / Problem-probe(약점탐침) / Experiment
> (후속실험)"의 머리글자다. (GPU 번호와 무관)

## 3. 결과 ① — MoE는 어떻게 라우팅하는가 (H1~H6)

| 가설 | 질문 | 결과 | 핵심 수치 |
|------|------|------|-----------|
| H1 | 도메인마다 다른 전문가를 쓰나? | ✅ 그렇다 | 같은 도메인 유사도 0.84 vs 다른 도메인 0.42 |
| H2 | 전문가 부하가 불균형한가? | ✅ 그렇다 | 지니계수 0.67, **32%는 거의 안 쓰임** |
| H3 | 깊은 층일수록 더 특화되나? | △ 평탄 | 초반≈후반, 특별한 깊이 의존 없음 |
| H4 | 라우터가 아슬아슬하게 고르나? | ✅ 그렇다 | 1·2등 점수 차이 margin **0.0023** (근소) |
| H5 | 모든 도메인이 공유하는 핵심 전문가? | ❌ 거의 없다 | 공유 core 단 **4.3%** |
| H6 | 라우팅만 보고 도메인을 맞힐 수 있나? | ✅ 매우 잘 | 도메인 분류 **99.8%** (찍으면 33%) |

**한 줄 해석**: 거대 MoE는 **도메인마다 거의 분리된 전문가 집합**을 쓴다(H1·H5·H6).
그리고 전문가의 약 **1/3은 거의 안 쓰인다**(H2). → "도메인별로 안 쓰는 전문가가
많다"는 압축의 실마리.

### 3.1 중요한 단서 — "집계 착시"
"수학은 전문가의 82%가 일치"처럼 들리지만, 이는 **200개를 모아서** 본 통계다.
개별 예시끼리 비교하면 top-8 중 **약 절반만** 겹친다(나머지 절반은 입력마다 유동).
→ 도메인→전문가 배정은 **결정적이지 않고 "분포적(통계적)"** 이다.
또한 같은 NLI라도 MNLI(36%) vs SNLI(64%)로 일관성이 2배 차이 → 라우팅 일관성은
"의미"뿐 아니라 **데이터셋이 얼마나 동질적인가**에도 좌우됨(해석 시 주의할 교란요인).

## 4. 결과 ② — 닫힌 문 vs 열린 문 (P1~P6)

처음 떠올린 압축 아이디어들이 실제로 통하는지 데이터로 검증했다.

| 가설 | 아이디어 | 결과 | 왜 |
|------|---------|------|----|
| P1 | 중복 전문가를 **병합**? | ❌ 안 됨 | 전문가들 출력 유사도 0.049 → 서로 다른 함수, 중복 없음 |
| P3 | 활성 전문가 **8→4로 축소**? | ❌ 안 됨 | k=4면 출력 오차 0.54 (큼) → 8개 다 필요 |
| P6 | 불안정한 **라우터를 다듬기**? | ❌ 불필요 | 무의미한 변형에도 top-8의 12%만 변동 → 이미 안정 |
| P2 | 라우팅이 단어에 끌려가나? | △ 부분적 | 같은 단어 47% vs 무작위 12% — 영향 있으나 전부는 아님 |
| **P4** | **첫 토큰이 전문가를 독점?** | ✅ **그렇다** | pos-0 일치 80%+ (E1에서 해부) |
| **P5** | **도메인 전용 전문가 존재?** | ✅ **그렇다** | 28%가 단일 도메인 전문가 → pruning은 "도메인 조건부" |

**한 줄 해석**: "중복 병합 / k 축소 / 라우터 손질"은 **세 문 다 닫혔다**(P1·P3·P6).
열린 문은 둘 — **도메인-조건부 가지치기(P5)** 와 **sink 라우팅(P4)**.

---

## 5. 결과 ③ — 열린 문 깊이 파기 (E1·E2·E3)

### 5.1 E1 — "sink 라우팅" 해부 (P4 후속)
**현상**: 문장의 **맨 첫 토큰**은 내용과 상관없이 거의 같은 전문가들로 간다.
- 위치 0의 예시 간 top-8 일치 **80.5%** (vs 깊은 위치 12%). 위치 1부터 급락.
- 이 "sink 전문가 집합"은 **도메인을 안 가린다**(수학/코드/NLI 교집합 74~79%).
- **그러나** 이 전문가들이 전체 토큰에서 차지하는 계산량은 **8.1%뿐**.
- → **현상은 진짜지만 압축 이득은 작다. 논문의 "보조" 발견으로 둠.**
- (참고: "attention sink"라는 알려진 현상의 MoE 라우팅 버전. 새롭고 흥미로움.)

### 5.2 E2 — 도메인별로 얼마나 버릴 수 있나 (상한 추정)
각 도메인의 라우팅 "질량"의 95%를 덮는 최소 전문가만 남기면 **층당 평균
math 50% / code 38% / nli 50%** 를 버릴 수 있다(상한). code가 가장 독립적.
→ 단, 이건 "라우팅 비중" 기준 상한일 뿐 **실제 정확도 보장은 아님**. 그래서 E3.

### 5.3 E3 — 실제로 꺼보고 정확도 확인 (핵심 검증) ✅
도메인이 안 쓰는 전문가를 실제로 **끄고**(라우터에서 -무한대 처리), 별도의
held-out 텍스트에서 perplexity(PPL, 낮을수록 좋음)와 다음-단어 정확도를 측정.

| 도메인 | 그대로(baseline) | **자기 도메인** 유지셋으로 16~22% 끔 | **틀린 도메인** 유지셋으로 끔 |
|--------|-----------------|------------------------------------|------------------------------|
| 수학 | PPL 10.83 | PPL +0.11 (거의 무손실) | 코드셋 적용 시 +1.08 |
| 코드 | PPL 10.91 | PPL +0.29 | 수학셋 +3.34 / NLI셋 +3.89 |
| NLI | PPL 32.81 | PPL +0.24 (거의 무손실) | **코드셋 +14.2 (붕괴)** |

**검증된 3가지 결론**:
1. **맞는 도메인으로 끄면 거의 무손실** — 16~22%를 꺼도 PPL 변화 +0.1~0.3.
   (99% 커버리지로 6~9%만 끄면 변화 ≈ 0, 노이즈 수준)
2. **틀린 도메인으로 끄면 무너진다** — PPL이 +1~**+14**까지 급등. 즉 *어느 전문가를
   끄느냐가 도메인마다 진짜 다르다*. 이게 "도메인-조건부"가 필수라는 직접 증거.
3. **코드는 민감** — 가장 독립적이라 여유가 적다(다음-단어 정확도 -7.6%p). 안전
   기본값은 "99% 커버리지(무손실)", 95%는 수학·NLI는 OK·코드는 주의.

## 6. 그래서 논문 문제는? (제안)

> **제목(가안)**: *Domain-Conditional Expert Pruning for Modern MoE LLMs*
>
> **문제 정의**: 거대 MoE는 도메인마다 거의 독립된 전문가 집합을 쓴다. 따라서
> **배포 도메인이 정해지면**, 그 도메인이 안 쓰는 전문가를 **정확도 손실 없이**
> 상당수 제거할 수 있다. 핵심 난점은 *무엇을 끄느냐가 도메인마다 달라서*, 도메인을
> 잘못 잡으면 성능이 붕괴한다는 점이다(측정으로 입증). → 이를 안전하고 자동으로
> 수행하는 **도메인-조건부 전문가 가지치기** 방법을 제안/검증한다.

**왜 이 문제가 좋은가 (measure-first의 강점)**:
- 흔한 압축 각도 3개가 *왜 안 되는지*를 데이터로 닫아, 논문의 동기가 단단하다(P1/P3/P6).
- 핵심 주장이 **추측이 아니라 측정+마스킹 실험으로 검증**됐다(H1·H5·E2·E3).
- "도메인 잘못 잡으면 +14 PPL"이라는 **명확한 실패 사례**가 있어 방법의 필요성이 자명.

**열린(=논문에서 풀) 질문**:
- 배포 시 도메인을 어떻게 **자동 판별**할까? (H6: 라우팅으로 99.8% 분류 가능이 힌트)
- 여러 도메인을 동시에 서빙할 때 유지셋을 어떻게 **합칠까**? (E2 겹침분석 활용)
- 코드처럼 민감한 도메인의 **안전 마진**을 어떻게 정할까?

## 7. 일반화 검증 — Mixtral 교차실험 (진행 중)

**우려**: 위 결과가 Qwen3 한 모델의 특이성일 수 있다.
**대응**: 구조가 다른 또 다른 대표 MoE인 **Mixtral-8x7B-Instruct**(32층, 층당
**8 전문가 top-2** — Qwen의 128/top-8과 대조적)로 **동일 실험을 재현 중**.
- GPU 8개 전부 사용, 모델 다운로드 후 H/P/E 전 실험 재실행 예정.
- 기대: "도메인-조건부 전문가 분리"가 **모델·아키텍처를 가로질러 성립**함을 보이면
  논문 주장이 단일 모델 일화가 아니라 **현대 MoE의 일반 성질**로 격상된다.
- *(이 절은 Mixtral 결과가 나오면 수치로 채워 갱신)*
